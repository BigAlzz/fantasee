1. Introduce a proper “Story State Graph”

Your document already notes that each scene depends on previous scenes, character bible, world rules and character state. That is the hidden dragon in the system. Right now, the architecture treats scenes as mostly independent, but narrative continuity is not truly independent. Scene 12 may depend on a wound from scene 4, a betrayal in scene 7, or a weapon introduced in scene 2.

I would add a Story State Graph as a first-class structure.

Instead of only:

Scene 1 -> Scene 2 -> Scene 3

You should track:

Character state
Object state
Location state
Relationship state
Plot promise state
Unresolved mystery state
Emotional arc state

For example:

{
  "character_states": {
    "Queen Elara": {
      "location": "The Forge",
      "injuries": ["burned left hand"],
      "emotional_state": "defiant but grieving",
      "known_secrets": ["the crown was forged from enemy steel"]
    }
  },
  "open_threads": [
    {
      "id": "thread-003",
      "promise": "The blacksmith knows who betrayed the queen",
      "introduced_scene": "scene-04",
      "status": "unresolved"
    }
  ]
}

This gives the critic and editor real memory. Without it, the critic is mostly a fancy proof-reader with a lantern. With it, the system becomes a continuity engine.

2. Add dependency-aware regeneration, not just scene-level regeneration

Your incremental approach says that changing scene 5 only regenerates scene 5 unless narrative changes cascade to the next scene. That is good, but I would make this more precise.

Use a dependency graph where every generated asset declares what it depends on.

Example:

{
  "asset": "scene-05-image-01",
  "depends_on": [
    "scene-05.visual_prompt",
    "scene-05.character_appearances",
    "story_bible.visual_style",
    "character.elara.costume"
  ]
}

Then if you change only the camera angle, only the affected images regenerate.
If you change Elara’s hair colour in the story bible, all images containing Elara are marked stale.
If you change narration text, only audio and subtitles regenerate.

This turns the app from “incremental-ish” into a proper creative compiler. Tiny edit in, only necessary outputs out.

3. Split “canonical story text” from “presentation text”

At the moment, each scene has narrative, narration, image prompt and subtitles. That is fine, but I would formalise the distinction.

Each scene should have:

Canonical scene event
What actually happens in the story.
Narration script
What the voice says.
Visual direction
What the image/video shows.
Audio performance direction
Voice, tone, pacing, emotion.
Continuity state update
What changed in the world after this scene.

This prevents a common generation problem: the image says one thing, the narration says another, and the plot state says a third. Little story goblins start chewing the furniture.

A better scene model would include:

{
  "canonical_event": "Elara reforges the broken crown and discovers the hidden inscription.",
  "narration_script": "In the dying light of the forge, Elara raised the broken crown...",
  "visual_brief": "Show Elara at the forge discovering glowing text inside the crown.",
  "state_delta": {
    "objects": {
      "crown": {
        "status": "reforged",
        "revealed_secret": "contains enemy oath"
      }
    }
  }
}

This would also make the editor and critic much more powerful.

4. Reduce the number of agents even further

The plan reduces Mindstream’s 20+ agents to about 8 focused agents, which is a good move. The current document already identifies Mindstream as over-engineered, with many agents doing little work .

I would go further and organise agents by decision type, not production step.

Suggested core agents:

1. Architect Agent
   Owns outline, story bible, structure, plot promises.

2. Scene Writer Agent
   Writes and rewrites canonical scene events and narration.

3. Visual Director Agent
   Creates image prompts, shot plans, style consistency.

4. Voice Director Agent
   Creates TTS directions and voice continuity.

5. Continuity Critic Agent
   Checks character, plot, world, timeline and unresolved threads.

6. Quality Critic Agent
   Scores drama, pacing, clarity, visuals, audio and emotional impact.

7. Repair Agent
   Converts critic findings into precise edits.

8. Render Agent
   Handles final packaging, video, subtitles and export.

The key is that each agent must have a non-overlapping responsibility. If two agents can edit the same field without rules, the system becomes a committee of caffeinated parrots.

5. Add a “change proposal” layer before edits are applied

Do not let agents directly overwrite scenes. Make them propose patches.

Example:

{
  "proposal_id": "repair-scene-07-v2",
  "reason": "Visual continuity issue: Elara is shown without the crown scar.",
  "changes": [
    {
      "path": "images[0].prompt",
      "old": "Queen Elara standing in moonlight...",
      "new": "Queen Elara standing in moonlight, the crown scar visible above her left brow..."
    }
  ],
  "affected_assets": ["scene-07-image-00"],
  "estimated_regeneration": {
    "images": 1,
    "tts": 0,
    "subtitles": 0
  }
}

Then the user can accept, reject, or edit the proposal. This gives you a beautiful human-in-the-loop workflow and avoids accidental AI vandalism.

6. Add a proper “Director’s Console” UI

The app should not only show a Netflix-style viewer. It needs a production dashboard.

I would add these views:

Story Board View

A grid of scenes with thumbnails, ratings, status and warnings.

Scene 01 ✅ 4.4
Scene 02 ⚠️ Visual mismatch
Scene 03 ✅ 4.1
Scene 04 ❌ Audio missing
Timeline View

Shows narration duration, image count, pacing and scene intensity.

Useful for spotting dead zones where five scenes in a row are slow and broody.

Continuity View

Shows characters, locations, objects and unresolved plot threads per scene.

Regeneration Preview

Before applying edits, show:

This change will regenerate:
- 1 image
- 0 audio files
- 0 subtitle files
Estimated affected scenes: scene-05 only
Critic Panel

A scorecard per scene:

Story: 4.2
Visual: 3.1
Audio: 4.5
Continuity: 2.8
Pacing: 3.7
Recommendation: Fix character appearance and tighten scene objective.

That makes the app feel like an editing suite, not only a story vending machine.

7. Use a database for index and metadata, but keep file-based manifests

Your proposed file structure is sensible: story files, scene manifests, version files and content-addressed assets. I would keep that.

But I would add SQLite for fast querying.

Use files for:

story.json
scene-01.json
scene versions
assets

Use SQLite for:

story index
asset index
generation jobs
critic scores
dependency graph
search
status
cache references

This gives you the best of both worlds: human-readable project files plus fast app behaviour.

Do not put everything into a JSON blob again. The document correctly identifies that Mindstream’s JSON blob storage is a weakness .

8. Add a job queue

The engine should not run long generation jobs directly inside API requests.

Add a local job queue such as:

RQ / Redis
Celery
Dramatiq
or even SQLite-backed jobs for local simplicity

Each job should have:

{
  "job_id": "job-123",
  "type": "generate_image",
  "story_id": "crown-of-steel",
  "scene_id": "scene-05",
  "status": "running",
  "progress": 65,
  "logs": [],
  "started_at": "...",
  "completed_at": null
}

This will make the app much more stable. Long image generation, TTS, Whisper and render jobs should be resumable, cancellable and visible.

9. Add “style locks”

One common weakness in AI story tools is visual drift. Scene 1 looks painterly, scene 2 looks photographic, scene 3 looks like a game loading screen, and by scene 9 the queen has changed species.

Add style locks at multiple levels:

Global style lock
Character appearance lock
Location style lock
Lighting palette lock
Camera language lock
Costume lock

Example:

{
  "visual_locks": {
    "global_style": "dark fantasy oil painting, cinematic, 16:9",
    "elara": {
      "hair": "silver braided hair",
      "scar": "thin scar over left eyebrow",
      "costume": "blackened steel armour with red cloak"
    },
    "forbidden_forge": {
      "lighting": "molten orange and blue shadow",
      "architecture": "ancient basalt arches"
    }
  }
}

The Visual Director should inject these into prompts automatically.

10. Improve the critic into a multi-pass diagnostic system

Your critic loop already identifies weak scenes and improves them, which is excellent . I would make the critic more structured.

Use separate passes:

Pass 1: Plot logic
Pass 2: Character continuity
Pass 3: Visual consistency
Pass 4: Emotional pacing
Pass 5: Narration quality
Pass 6: Audio/subtitle technical QA
Pass 7: Audience impact

Then produce machine-actionable findings:

{
  "scene_id": "scene-12",
  "issue_type": "continuity",
  "severity": "high",
  "field": "narration.text",
  "problem": "The scene says the crown is lost, but scene 10 marked it as recovered.",
  "suggested_fix": "Change the line to indicate the crown is hidden, not lost."
}

This makes the Repair Agent far more precise.

18. Add provider abstraction properly

Your current stack uses Xiaomi/MiMo, ComfyUI, Whisper and FFmpeg. The architecture already proposes centralising LLM, TTS and generator wrappers .

I would formalise provider interfaces:

class LLMProvider:
    def generate_json(self, prompt, schema): ...

class ImageProvider:
    def generate_image(self, image_request): ...

class TTSProvider:
    def generate_audio(self, tts_request): ...

class SubtitleProvider:
    def align(self, audio, text): ...

Then you can swap:

MiMo -> LM Studio -> DeepSeek -> OpenAI-compatible endpoint
ComfyUI -> Stability -> Replicate -> local SD
MiMo TTS -> Kokoro -> ElevenLabs -> Piper
Whisper -> faster-whisper -> API transcription

This protects the engine from provider changes.

19. Add schema-first LLM output

The document mentions brittle JSON parsing and structured scene outline parsing issues in Fantasee .

This is a must-fix.

Every LLM output should use strict Pydantic validation:

class SceneOutline(BaseModel):
    scene_id: str
    title: str
    canonical_event: str
    emotional_purpose: str
    characters_present: list[str]
    location: str
    plot_threads_opened: list[str]
    plot_threads_closed: list[str]

If validation fails, send the model a repair prompt with the validation error.

Never silently parse broken output. Silent failure is how pipelines become haunted furniture.

20. My recommended revised architecture

I would structure it like this:

STORY ENGINE

1. Project Layer
   - Story metadata
   - Versions
   - Branches
   - Render recipes

2. Canon Layer
   - Story bible
   - Character bible
   - World rules
   - Plot threads
   - Story state graph

3. Scene Layer
   - Canonical event
   - Narration script
   - Visual brief
   - Voice direction
   - State delta

4. Asset Layer
   - Images
   - Audio
   - Subtitles
   - Video clips
   - Content-addressed cache

5. Pipeline Layer
   - Dependency graph
   - Incremental regeneration
   - Job queue
   - Checkpoints

6. Intelligence Layer
   - Architect
   - Scene writer
   - Visual director
   - Voice director
   - Critic
   - Repair agent

7. App Layer
   - Director console
   - Storyboard
   - Timeline
   - Critic dashboard
   - Regeneration preview
   - Viewer
Biggest priority list

I would prioritise these first:

Pydantic schema-first models
This fixes brittle parsing and gives the whole engine a spine.
Story State Graph
This upgrades continuity from vibes to machinery.
Dependency graph for regeneration
This makes incremental generation truly reliable.
Job queue and progress tracking
This prevents long-running generation from making the app feel frozen.
Director’s Console UI
This turns the system into a real creative tool.
Change proposals before applying AI edits
This keeps humans in control and prevents unwanted rewrites.
Style locks and character locks
This improves visual consistency across scenes.

My overall view: the architecture is already strong, but the next evolution should be to treat the engine less like a pipeline and more like a non-linear creative compiler: story state goes in, dependencies are tracked, only stale outputs regenerate, and every creative decision can be inspected, scored, patched and rolled back.
